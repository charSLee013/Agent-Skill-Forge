# Structured Object Map

Use a structured object map when the learning subject has internal structure: a repository, system, process, protocol, design, paper, workflow, or toolchain.

## Rule

Do not organize the main student narrative by filesystem tree, document order, or artifact list unless that structure is itself the concept being taught.

First derive the subject's learning structure.

## Map Dimensions

Include the dimensions that fit the subject:

- Purpose: what problem it exists to solve.
- Interfaces: inputs, outputs, commands, APIs, user actions, or contracts.
- Representation: data, terms, objects, states, formats, or schemas.
- Process: control flow, lifecycle, sequence, procedure, or transformation.
- State: initial state, transitions, memory, persistence, or checkpoints.
- Constraints: invariants, assumptions, limits, policies, or compatibility rules.
- Environment: dependencies, runtime context, tools, deployment, or operating conditions.
- Verification: tests, metrics, examples, checks, or observable outcomes.
- Unknowns: undisclosed, unverified, stale, or ambiguous facts.

## Evidence Placement

Implementation paths, line numbers, symbols, section anchors, or command outputs can support the map, but they should not become the main explanation.

Use detailed anchors in:

- Source matrix.
- Collapsed evidence appendix.
- Backstage truth map.
- Reference page for module or artifact lookup.

## Gate Questions

Before writing student pages, answer:

- What are the core claims?
- What are the subject's main interfaces?
- What representations or states must the learner understand?
- What processes or transformations matter?
- What constraints or invariants shape behavior?
- What failure modes or unknowns matter?
- What needs visible evidence?
- What needs a mechanism explanation?
